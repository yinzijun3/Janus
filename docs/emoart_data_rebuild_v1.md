# EmoArt Data Rebuild v1

## Summary

This note defines the first full-dataset rebuild for the current JanusFlow brushstroke program. The goal is to stop using the old texture-rich subset as the default source of truth and instead build a style-balanced master manifest plus two continuation subsets:

- `train_full_balanced_v1`: preserve subject semantics and composition
- `train_texture_balanced_v2`: push visible brushwork without losing head-sensitive cases
- `train_portrait_structure_v1`: structure-preserving recovery stage for portrait-dominant styles

The rebuild is motivated directly by the observed failure pattern from E8 through E16:

- visible brushstroke strength is possible
- portrait and baroque failures are still the first place where stronger texture routes break
- the old `train_texture_rich_v1` subset over-compresses the data into a texture-first slice and under-represents several failure-critical styles

## Method

The rebuild is generated by `build_emoart_full_rebuild_v1.py` from `annotation.official.json`.

Per-sample fields include:

- official metadata: `request_id`, `image_path`, `description`, `style_name`
- extracted text slots: `content_text`, `brushstroke_text`, `line_quality_text`, `composition_text`, `color_text`, `emotional_impact`
- prompt metadata: `texture_tags`, `medium_tags`
- three explicit prompts: `prompt_v2_content_safe`, `prompt_v2_texture_push`, `prompt_v2_portrait_safe`
- scoring and risk fields: `texture_score`, `frame_risk_score`, `portrait_likelihood`, `structure_risk_score`, `head_sensitive`, `style_portrait_dominant`

The scoring policy is intentionally split:

- `texture_score` comes only from textural cues plus image high-frequency statistics
- `frame_risk_score` stays independent and is used for filtering or ranking only
- `structure_risk_score` is driven by portrait-like text cues and central-subject composition cues

## Output Layout

Outputs are written to:

`/root/autodl-tmp/data/emoart_5k/gen_full_official_rebuild_v1`

Important files:

- `master_manifest.jsonl`
- `train_full_balanced_v1.jsonl`
- `val_balanced_v1.jsonl`
- `train_texture_balanced_v2.jsonl`
- `train_portrait_structure_v1.jsonl`
- `style_summary.json`
- `subject_summary.json`
- `data_card.md`

## Commands

```bash
python build_emoart_full_rebuild_v1.py

python train_janusflow_art.py \
  --config configs/janusflow_art/brushstroke/brush_spatial_hf_head_full_balanced_v1.yaml \
  --max-steps 1 \
  --skip-final-eval

python train_janusflow_art.py \
  --config configs/janusflow_art/brushstroke/brush_spatial_hf_head_texture_balanced_v2.yaml \
  --resume-from-checkpoint /root/autodl-tmp/emoart_gen_runs/brushstroke_experiments/D1_brush_spatial_hf_head_full_balanced_v1/final_checkpoint
```

## Acceptance

- every official annotation row is either materialized into `master_manifest.jsonl` or listed in `bad_samples.jsonl`
- `val_balanced_v1` keeps `10` samples per style
- `train_texture_balanced_v2` keeps `28` samples per style
- `train_portrait_structure_v1` contains only `head_sensitive` samples from portrait-dominant styles
- derived training manifests set `rendered_prompt` explicitly so JanusFlow-Art uses the prompt variant exactly as written
